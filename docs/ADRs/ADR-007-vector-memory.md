# ADR-007: Vector Memory para Aprendizado Contínuo

## Status
✅ **Aceito** (2025-09-29)

## Contexto

O sistema BuildToFlip Central de Inteligência Jurídica (Standard Upgrade - Onda 2.2) necessita de **memória de longo prazo** para:

1. **Evitar Reprocessamento:** queries similares não devem exigir trabalho redundante.
2. **Contextualização:** decisões futuras precisam considerar interações passadas.
3. **Aprendizado Contínuo:** o agente deve melhorar performance com o tempo.
4. **Redução de Custos:** minimizar chamadas de LLM desnecessárias.

Sem memória persistente:

- Cada query é processada do zero (~800ms).
- Histórico de uso não é aproveitado.
- Sessões independentes não compartilham contexto.
- Custos de API crescem linearmente.

## Decisão

Implementar **Vector Memory** usando:

- **Vector DB:** ChromaDB 0.5.0 (HTTP server em Docker).
- **Embeddings:** OpenAI `text-embedding-3-small` (1536 dimensões).
- **Persistência:** volume Docker `chromadb_data`.
- **Client:** `chromadb-client` (Python SDK).

### Arquitetura

```
┌─────────────────────────────────────────────────┐
│              USER INPUT                         │
└───────────────┬─────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────────┐
│  ❶ MEMORY RECALL (ChromaDB)                    │
│     • Query: text-embedding-3-small             │
│     • Returns: K=3 most similar past tasks      │
│     • Latency target: <200ms                    │
└───────────────┬─────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────────┐
│  ❷ TASK PROCESSING (já existente)              │
│     • Intent Classification                     │
│     • Tribunal Delegation                       │
│     • Result Aggregation                        │
└───────────────┬─────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────────┐
│  ❸ MEMORY STORAGE (persistir)                  │
│     • Document: task + result + metadata        │
│     • Embedding: auto-generated by OpenAI       │
│     • Storage: ChromaDB volume (persistente)    │
└─────────────────────────────────────────────────┘
```

### Implementação

**Ciclo de Vida:**

1. **Recall:** antes de processar, buscar K=3 memórias similares.
2. **Process:** executar tarefa (possível uso de cache se similaridade alta).
3. **Remember:** armazenar interação + embedding no ChromaDB.

**Metadata Armazenado:**

```python
{
    "tribunals": ["TJSP"],
    "intent_operacao": "status_check",
    "intent_confidence": 0.95,
    "execution_time": 0.8,
    "timestamp": "2025-09-29T20:00:00Z",
    "result_snapshot": "{...}"  # usado para cache rápido
}
```

## Alternativas Consideradas

1. **Redis com Cache Simples**
   - ✅ Zero setup adicional
   - ❌ Sem matching semântico
   - ❌ Não oferece embeddings

2. **Pinecone (SaaS Vector DB)**
   - ✅ Escalável, gerenciado
   - ❌ Custo recorrente e lock-in

3. **PostgreSQL + pgvector**
   - ✅ Alinhado a stack futura
   - ❌ Setup mais complexo e desnecessário para MVP

4. **ChromaDB** ✅ **Escolha Final**
   - ✅ Open source, setup via Docker
   - ✅ Integração direta com OpenAI embeddings
   - ✅ Persistência via volume
   - ⚠️ Single instance (sem HA nativo)

## Consequências

### Positivas ✅

1. **Performance:** redução de 20-30% na latência de queries repetidas via cache hit.
2. **Custos:** menor uso de LLMs graças a reutilização de resultados.
3. **UX:** respostas contextualizadas aumentam confiança do usuário.
4. **Observabilidade:** métricas de memória expostas no supervisor.

### Negativas ⚠️

1. **Complexidade:** novo serviço Docker + dependências opcionais.
2. **Dependência externa:** exige OPENAI_API_KEY configurada.
3. **Latência adicional:** recall adiciona ~150ms por consulta (aceitável dentro da meta).
4. **Crescimento de storage:** ~10MB/dia, requer manutenção periódica.

### Mitigações 🛡️

- Fallback graceful quando ChromaDB/OpenAI indisponíveis.
- Monitoramento via Prometheus + logs do ledger.
- Script `scripts/validate-wave2.2.sh` para gates de qualidade.
- Limpeza trimestral de memórias antigas (>90 dias) planejada.

## Métricas de Sucesso

| Métrica | Target | Como medir |
| --- | --- | --- |
| Recall Precision | >75% | % de memórias relevantes entre recalls |
| Recall Latency P95 | <200ms | Tempo de consulta ao ChromaDB |
| Learning Effect | >10% | Redução de latência entre 1ª e 2ª consulta |
| Cache Hit Rate | >40% | % de tarefas com `cache_hit=True` |
| Persistência | 100% | Memórias sobrevivem a restart |

## Validação

- ✅ Arquitetura aprovada pelo time técnico.
- ✅ Integração com SupervisorAgent (recall + cache + remember).
- ✅ Testes de integração e emergentes cobrindo ciclo completo.
- ✅ Script `validate-wave2.2.sh` garante pré-requisitos e benchmarks.
